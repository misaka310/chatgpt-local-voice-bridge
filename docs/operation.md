# 操作と検証

## 通常の使い方

1. `LocalVoiceBridge.exe`を起動します。
2. 通知領域で`Status: Ready`または`Status: Ready (existing)`を確認します。
3. デスクトップペットを左ダブルクリックするか、通知領域の`Show Local Voice panel`を選びます。
4. Windows Local Voice小窓で必要に応じて`Ref`と`Volume`を設定し、`Auto`をオンにします。
5. Chrome / Braveで開いている任意のChatGPTタブへ新しいメッセージを送ります。
6. 新しい返答の冒頭プレビューが共通キューの順番で一度だけ再生されることを確認します。

Chrome / Brave内にはLocal Voice操作パネルを表示しません。日常操作はChrome外のWindows小窓だけで行います。小窓には`Ref`、`Volume`、`マイク会話`、`Auto`、`Next`、`Regen`、`Replay`、状態、現在文章、キュー数、登録済みChatGPTタブ数を表示します。読み上げ範囲、文字起こしモデル、送信前猶予は拡張機能アイコンを右クリックして`オプション`から設定します。

小窓は常に手前へ表示されます。×を押すと終了ではなく非表示になります。上部を左ドラッグすると移動でき、位置は次回起動時に復元されます。

## マイク会話モード

`マイク会話`は、Chrome版ChatGPTのモデル、Web検索、ログイン済み会話履歴をそのまま使いながら、入力だけをローカル音声認識へ置き換える任意モードです。既存の読み上げ専用利用ではオフのままにしてください。右Ctrl単独では録音せず、通常のCtrl操作へ干渉しません。

### 準備

1. `setup-voice-env.cmd`を開き、`読み上げ + マイク会話`プロファイルを実行して`faster-whisper`と`sounddevice`を導入します。
2. Windowsの設定でデスクトップアプリのマイクアクセスを許可し、既定の入力デバイスを選びます。
3. Chrome / Braveで拡張機能を再読み込みし、ログイン済みのChatGPT会話画面を開きます。
4. Chrome / Braveの拡張機能アイコンを右クリックして`オプション`を開き、文字起こしモデルと送信前猶予を設定します。その後、Windows Local Voice小窓で`マイク会話`をオンにします。選択したSTTモデルはこの時点で準備され、未取得なら`local-api/runtime/stt-models/`へダウンロードします。準備完了までは録音を開始しません。

初期STTモデルは`small`です。拡張機能の`オプション`で`medium`と`large-v3-turbo`へ変更できます。モデル変更時も先に準備状態へ入り、準備完了後に待機中へ戻ります。まずCUDAの`float16`を試し、利用できない場合はCPUの`int8`へ自動フォールバックします。実際に使用されたデバイスは小窓へ`CUDA`または`CPU fallback`として表示されます。

### 操作

1. 入力先にしたいChatGPTタブの入力欄へフォーカスを入れます。
2. 右Ctrlを押したまま、右Shift左にある`＼ / _`キーを押すと、再生中のローカルTTSと未再生キューを破棄します。
3. 2キーを押している間だけ録音します。右Ctrl単独、左Ctrl、`＼ / _`キー単独、キーリピートでは録音を開始しません。
4. どちらか一方を離すと録音を終了し、日本語をローカルで文字起こしします。録音トリガーとして押した`＼ / _`は入力欄へ挿入しません。
5. 送信先は録音開始時に最後にフォーカスしていたChatGPT入力欄へ固定されます。文字起こし中に別タブへ移動しても変わりません。
6. 結果が空でなく、対象のChatGPT入力欄も空なら、今回の文字列をReact側へ反映される入力イベント付きで挿入します。
7. 初期値0.7秒の間にEscを押すと、今回挿入した文字列だけを削除して送信を中止します。猶予は拡張機能の`オプション`で0.0〜5.0秒へ変更でき、0秒では即送信します。
8. Escを押さなければ、有効になったChatGPTの送信ボタンを1回だけ押します。
9. 返答は既存Auto経路で検出され、ローカルTTSで読み上げます。録音・文字起こし・送信準備中に他タブへ返答が来ても割り込み再生せず、その返答は共通キューへ入れません。読み上げ中に同じ2キーを押すと、その返答の残りも破棄し、次の録音へ移れます。

対応版のYouTube Dictation Pause Controlが起動している場合、2キーが録音開始として確定した時点で`source=local-voice-bridge, active=true`を、どちらかを離した時点で`active=false`を`http://127.0.0.1:17654/state`へ送ります。通知はキーボードフックとは別スレッドで行い、通知先が停止中でも録音処理は中断しません。別ポートを使用する場合は`YOUTUBE_DICTATION_PAUSE_STATE_URL`を設定します。通知先は`http`のloopbackホストと`/state`に限定されます。

入力欄に既存文章がある場合、入力欄または送信ボタンを検出できない場合、ページ遷移が発生した場合は自動送信しません。既存文章を上書きまたは削除せず、小窓へ理由を表示します。

### 状態表示

小窓には`オフ`、`モデル準備中`、`待機中`、`録音中`、`文字起こし中`、`Escキャンセル待ち`、`送信中`、`ChatGPT応答待ち`、`読み上げ中`、`エラー`を表示します。マイク権限、入力デバイス、STT依存関係、モデル、ChatGPT DOMの問題はエラー欄に具体的な確認先を表示します。

### ローカルデータと制約

録音はメモリ上だけで処理し、音声ファイルと文字起こし履歴を保存しません。loopbackイベントも1回取得された後に破棄され、永続状態ファイルへ会話本文を含めません。Git管理外になるのはSTTモデル、録音・文字起こし用runtime領域、生成音声、参照音声、Chromeプロファイル、ログ、認証情報です。

TTSの生成音声は再生・Replay用に一時保存しますが、起動時と生成後に自動整理します。初期保持上限は最新1,000件、合計1GB、14日以内です。通知領域の`Clear generated audio...`は生成音声だけを削除し、参照音声、設定、STTモデルには触れません。

既知の制約は、Windowsの既定マイクだけを使用すること、ChatGPT入力欄と送信ボタンのDOM変更に影響されること、初回モデル取得にネットワークと空き容量が必要なことです。CPUフォールバックは動作継続用で、CUDAより文字起こし待ち時間が長くなります。TTS側のIrodori v3は従来どおりNVIDIA GPUが必要です。

## 各ボタン

- `Auto`: オンにした後で、開いているすべてのChatGPTタブへ新しく表示されたassistant返答を対象にします。オンにする前から表示されていた返答と、`思考中`・`考え中`・`Thinking`・`画像を分析しています`だけの途中状態は読みません。返答全文ではなく、冒頭プレビューだけを一度再生します。初期値は最大2行・80文字で、拡張機能の`オプション`から変更できます。
- `Next`: 直近の手動操作対象となる返答の次のチャンクへ進みます。
- `Regen`: 現在のチャンクを同じ設定で再生成します。
- `Replay`: 再生成せず、直前に生成済みの音声をもう一度再生します。

Autoの読み上げ対象は最後に触った1タブだけではありません。開いている全ChatGPTタブです。各タブから届いた返答は1つの共通キューへ入り、順番に再生されます。

## 役割の分担

- `LocalVoiceBridge.exe`: Windows常駐アプリを起動
- Windows Local Voice小窓: 日常の音声操作と状態表示
- 通知領域: 小窓の表示、状態確認、再起動、フォルダ表示、自動起動、再セットアップ、終了
- Chrome / Brave拡張: 各ChatGPTタブの返答検出と音声再生
- Chrome / Brave拡張の`オプション`: 読み上げ範囲、文字起こしモデル、送信前猶予の設定
- デスクトップペット: 表示、左ドラッグ移動、左ダブルクリックによる小窓の開閉

ペットのシングルクリックと右クリックでは何も起きません。ドラッグ直後のダブルクリックは誤操作防止のため無効になります。

## GPU不要のデモ

```bat
npm ci
npx playwright install chromium
npm run demo
```

Node.jsとChromiumだけで安全なローカルフィクスチャを開きます。Python、CUDA、GPU、モデル、ChatGPTログインは不要です。実際の拡張機能コードと`scripts/mock-voice-api.js`を使用します。Chromiumを閉じるか`Ctrl+C`を押すと、mock APIと一時ブラウザプロファイルを片付けます。

デモではWindows小窓そのものではなく、同じloopback control APIを使って外部操作と拡張機能の境界を検証します。

## 自動テスト

```bat
npm run test:ci
```

`test:ci`は次を検証します。

- APIと音声URLが127.0.0.1のloopbackだけを使用すること
- Chrome内にLocal Voice操作パネルやペット本体を生成しないこと
- Windows小窓の設定・位置・表示状態が管理できること
- ペットの左ダブルクリックで小窓を開閉し、左ドラッグ後は誤開閉しないこと
- 外部小窓の`Ref`、`Volume`、`Auto`が拡張機能へ同期されること
- `Ref`がデスクトップペットAPIへ同期され、空または`none`は`placeholder`になること
- Autoをオンにする前から表示されていた返答へ`POST /v1/speak`を送らないこと
- Auto後の新しい返答だけを対象にし、`思考中`などの途中状態を送信しないこと
- 20文字未満の短い返答も安定後に読み上げること
- `Next`、`Regen`、`Replay`のコマンドを1回だけ実行すること
- 複数のChatGPTタブすべてが1つの共通Autoキューへ追加されること
- 外部小窓へ現在文章、キュー数、再生状態、登録タブ数が同期されること

mockの成功は、実Irodoriの音質、GPU使用、将来の実ChatGPT DOMを保証しません。

## 実機Irodoriの確認

```bat
npm run test:e2e:real
```

先に`setup-voice-env.cmd`を完了し、NVIDIA GPU、CUDA、モデルを用意してください。テストは専用loopbackポートと一時状態ファイルを使い、普段の8717番やペット設定へ干渉せずに次を確認します。

- 外部Autoによる実際の`/v1/speak`と生成音声取得
- 外部`Next`による次チャンク再生
- 実Ref、Chrome保存値、デスクトップペット選択の一致
- 2つのChatGPTタブから同じ共通キューへ追加されること

ターミナルでサーバーログを直接確認する場合だけ`run-voice-stack.cmd`を使います。通常利用の起動入口ではありません。
